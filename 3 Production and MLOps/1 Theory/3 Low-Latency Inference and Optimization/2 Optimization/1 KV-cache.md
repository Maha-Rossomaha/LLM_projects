# KV-кэш: память, управление и оптимизация

## 1. Определение и мотивация

### 1.1 Зачем нужен KV-кэш

LLM генерируют текст авторегрессивно: на каждом шаге модель получает на вход все предыдущие токены и предсказывает следующий. Без кэша каждый шаг требовал бы пересчёта attention для всей последовательности заново — сложность $O(L^2)$ на шаг, где $L$ — длина контекста.

**KV-кэш (key-value cache)** — это сохранённые матрицы Key и Value из каждого слоя трансформера для всех уже обработанных токенов. На шаге $t$:

- Новый токен $x_t$ проходит через все слои.
- В каждом слое вычисляются $K_t$ и $V_t$ для этого токена.
- $K_t$ и $V_t$ **добавляются** в кэш (конкатенация с предыдущими).
- Attention считает только по новому токену: $Q_t \cdot [K_1, ..., K_t]^T$.

Результат: сложность одного decode-шага — $O(L)$, а не $O(L^2)$.

### 1.2 Цена кэширования

KV-кэш жертвует памятью ради вычислительной эффективности. Для LLaMA-7B:

- Один токен в кэше: $2 \times 32 \text{ слоя} \times 32 \text{ головы} \times 128 \text{ dim} \approx 262\text{K} \text{ значений} \approx 0.5\text{ MB}$ (в bfloat16).
- На 4K контекст: $\approx 2\text{ GB}$ на одну последовательность.
- На 256 параллельных запросов c контекстом 4K: $\approx 500\text{ GB}$ — больше, чем помещается на одной H100 (80 GB).

> KV-кэш — доминирующий потребитель GPU-памяти при инференсе LLM. Веса модели — константа. Кэш растёт линейно с числом запросов и длиной контекста.

## 2. Как устроен KV-кэш

### 2.1 Структура

Для одного слоя трансформера:

```python
# Одна голова attention
key_cache[layer][head] = Tensor[seq_len, d_head]   # Key
value_cache[layer][head] = Tensor[seq_len, d_head]  # Value
```

В памяти всё хранится как один непрерывный тензор на слой:

```python
# Реальный формат (объединённый на все головы)
key_cache[layer] = Tensor[seq_len, num_heads * d_head]  # или с GQA: num_kv_heads * d_head
value_cache[layer] = Tensor[seq_len, num_heads * d_head]
```

При GQA (Grouped-Query Attention) количество хранимых голов сокращается:

| Модель | num_heads | num_kv_heads | Экономия KV-кэша |
|--------|-----------|-------------|------------------|
| LLaMA-2-7B | 32 | 32 | 1× (без GQA) |
| LLaMA-2-70B | 64 | 8 | 8× против полных head |
| Mistral-7B | 32 | 8 | 4× |
| LLaMA-3-70B | 64 | 8 | 8× |

### 2.2 Prefill vs Decode

KV-кэш наполняется в два этапа:

**Prefill (первый forward pass):**

- Промпт длины $P$ обрабатывается за один проход.
- Attention считается для всех пар токенов сразу: $Q \cdot K^T$, где $Q, K \in \mathbb{R}^{P \times d}$.
- KV-кэш заполняется $P$ токенами за один шаг.
- **Compute-bound:** матричное умножение доминирует, GPU загружен.

**Decode (генерация):**

- Каждый шаг обрабатывает ровно один новый токен.
- Attention только между новым $Q_t$ и всеми $K_{1..t}$.
- KV-кэш растёт на 1 токен за шаг.
- **Memory-bound:** основной лимит — пропускная способность HBM, а не FLOPs.

```python
# Упрощённая схема
class KVCache:
    def __init__(self, num_layers, num_heads, head_dim, max_seq_len, dtype):
        # Предварительная аллокация: [num_layers, 2, max_seq_len, num_heads, head_dim]
        self.cache = torch.zeros(num_layers, 2, max_seq_len, num_heads, head_dim, dtype=dtype)
        self.seq_len = 0

    def prefill(self, layer, key, value):
        """Заполнение кэша на prefill-этапе (P токенов за раз)."""
        p = key.shape[2]  # длина промпта
        self.cache[layer, 0, :p] = key
        self.cache[layer, 1, :p] = value
        self.seq_len = p

    def append(self, layer, key, value):
        """Добавление одного токена на decode-шаге."""
        t = self.seq_len
        self.cache[layer, 0, t] = key
        self.cache[layer, 1, t] = value
        self.seq_len += 1

    def get(self, layer):
        """Получение всего кэша для слоя."""
        return self.cache[layer, 0, :self.seq_len], self.cache[layer, 1, :self.seq_len]
```

### 2.3 Жизненный цикл в continuous batching

При continuous batching каждый запрос имеет собственный KV-кэш, который живёт от prefill до завершения генерации:

```
t=0: Prefill(A)          → cache_A = [prompt_A]
t=1: Decode(A) + P(B)    → cache_A = [prompt_A, t1], cache_B = [prompt_B]
t=2: Decode(A,B) + P(C)  → cache_A = [prompt_A, t1, t2], cache_B = [prompt_B, t2], cache_C = [prompt_C]
t=3: A завершён, cache_A освобождён
     Decode(B,C) + P(D)  → cache_D = [prompt_D]
```

Проблема: если кэш выделяется непрерывным блоком с запасом (`max_seq_len`), освобождение запроса A создаёт «дыру», которую нельзя переиспользовать для запроса D (фрагментация). Решение — блочное управление (PagedAttention, ZeRO KV Cache).

## 3. Оптимизация KV-кэша

### 3.1 PagedAttention (блочное управление)

Основная идея — KV-кэш разбивается на блоки фиксированного размера (16–64 токена). Block table связывает логические позиции последовательности с физическими блоками:

```
Логическая последовательность A длиной 73 токена:
  ┌────┬────┬────┬────┬────┐
  │ b0 │ b1 │ b2 │ b3 │ b4 │  (5 блоков по 16 токенов, последний неполный)
  └────┴────┴────┴────┴────┘

Block table:
  seq_A → [phys_block_7, phys_block_3, phys_block_12, phys_block_1, phys_block_19]

Физическая память:
  [b7: A][b3: A][b12: A][b1: A][b19: A][b2: B][b5: B]...
```

Преимущества:

| Подход | Фрагментация | Выделение | Макс. длина |
|--------|-------------|-----------|-------------|
| Непрерывный кэш | Внешняя (дыры) | С запасом, pre-allocate | Фиксирована |
| PagedAttention | Нет (любой свободный блок) | On-demand | Ограничена только общей памятью |

**Дополнительно:** block table позволяет разделять блоки между последовательностями (prefix caching). Если два запроса начинаются одинаково, блоки с их общим префиксом имеют ref_count = 2 и не удаляются, пока активна хотя бы одна последовательность.

### 3.2 KV-quantization (сжатие кэша)

KV-кэш обычно хранится в bfloat16/float16 — 2 байта на значение. Можно сжимать:

| Формат | Байт на значение | Экономия | Потери качества |
|--------|-----------------|----------|-----------------|
| FP16 | 2 | 1× | — |
| INT8 | 1 | 2× | Минимальные |
| INT4 | 0.5 | 4× | Заметные на длинном контексте |
| FP8 (H100) | 1 | 2× | Практически нет |

```python
# INT8 квантование KV-кэша (per-token)
def quantize_kv(key_fp16):
    scale = key_fp16.abs().max(dim=-1, keepdim=True).values
    key_int8 = (key_fp16 / scale * 127).round().to(torch.int8)
    return key_int8, scale  # scale — для деквантования

def dequantize_kv(key_int8, scale):
    return key_int8.float() * scale / 127
```

**Нюанс:** квантизация K и V влияет на attention по-разному. K участвует в вычислении $QK^T$ — квантизация K вносит шум в softmax. V участвует в взвешенном суммировании — квантизация V влияет на итоговое представление. На практике V менее чувствителен к квантизации, чем K.

### 3.3 Prefix caching (кэш общих префиксов)

Если несколько запросов начинаются с одинакового префикса (системный промпт, история чата), KV-кэш префикса вычисляется один раз и разделяется:

```
Запрос 1: [SYS] You are a helpful assistant. [USER] What is Python?
Запрос 2: [SYS] You are a helpful assistant. [USER] Explain attention.
              └────── общий префикс ──────┘
```

Блоки с KV-кэшем префикса помечаются как shared (ref_count > 1). Пока хотя бы одна последовательность ссылается на блок, он не освобождается.

**Ограничение:** кэш работает только при точном совпадении токенов. Если префиксы различаются хотя бы на один токен — кэш не переиспользуется.

### 3.4 CPU offload

При нехватке GPU-памяти блоки KV-кэша можно выгружать в CPU (или даже NVMe):

```
t=0: KV-кэш полностью на GPU
t=50: GPU-память заканчивается → evict старых блоков на CPU
t=51: Нужен блок, который на CPU → загрузить обратно на GPU
```

**Проблема:** PCIe — узкое место. Загрузка 1 GB с CPU на GPU через PCIe 4.0 ×16 занимает ~30 мкс (теоретически), реально — 50–200 мкс с учётом overhead. Если во время decode нужен выгруженный блок, latency резко возрастает.

Offload — крайняя мера, когда без неё модель не помещается. Для production-сценариев с SLA лучше уменьшить `max_seq_len` или `max_batch_size`.

### 3.5 Влияние GQA

GQA (Grouped-Query Attention) уменьшает количество хранимых голов K и V в $n_q / n_{kv}$ раз:

| Модель | Размер KV-кэша (4K контекст) | С GQA |
|--------|------------------------------|-------|
| LLaMA-2-7B (32 KV голов) | ~2 GB | — |
| LLaMA-2-70B (8 KV голов) | ~10 GB | — |
| Гипотетическая 70B с полными heads | ~80 GB | 8× экономия |

GQA — аппаратное решение: модель изначально обучена с меньшим числом KV-голов. На уже обученную модель GQA не накладывается.

### 3.6 Сравнение методов

| Метод | Экономия памяти | Влияние на latency | Влияние на качество |
|-------|----------------|-------------------|-------------------|
| PagedAttention | Устраняет фрагментацию | Нет (overhead block table) | Нет |
| KV-quantization (INT8) | 2× | Небольшое (деквантование) | Минимальное |
| Prefix caching | Зависит от префикса | Нет | Нет |
| CPU offload | Неограниченная | Резкий рост при evict | Нет |
| GQA | 2–8× | Уменьшение | Минимальное (заложено архитектурой) |

## 4. Практические следствия

### 4.1 KV-кэш как лимитирующий фактор

При continuous batching максимальный batch size часто ограничен не compute, а памятью под KV-кэш:

```python
# Расчет максимального batch
gpu_memory = 80_000  # MB, H100
model_weights = 35_000  # MB, LLaMA-70B в FP16
kv_per_token = 2.6  # MB/токен, LLaMA-70B
avg_seq_len = 2048  # средняя длина контекста
kv_per_seq = kv_per_token * avg_seq_len  # ~5.3 GB

available = gpu_memory - model_weights  # ~45 GB
max_batch = int(available / kv_per_seq * 0.9)  # ~8 запросов
```

На 8 запросах GPU compute недогружен, но больше запросов не помещается. Решение — KV-quantization или GQA.

### 4.2 Memory-bound decode

95% времени decode-шага тратится не на вычисления, а на загрузку:

1. **Веса модели** — из HBM в SRAM для каждого слоя.
2. **KV-кэш** — все предыдущие K для вычисления $Q \cdot K^T$.

Для 70B модели на decode-шаге загружается:
- Веса: $\approx 2 \times 70\text{B} \times 2\text{ B} = 140\text{ GB}$ (все слои за один decode).
- KV-кэш: $2 \times 80 \times 8 \times 128 \times 2 \times 4096 \approx 1.3\text{ GB}$ (80 слоёв, 8 KV-голов, 4K контекст).

При пропускной способности H100 HBM (3.35 TB/s): decode latency $\approx 140 / 3350 \approx 42\text{ ms}$ без учёта кэша — и это на один слой за раз, с учётом pipelining. Загрузка KV-кэша добавляет ещё 5–10%.

### 4.3 Attention sink

На длинных контекстах первые токены (особенно BOS) получают непропорционально большое внимание — это называется **attention sink**. KV-кэш для этих токенов всегда «горячий» (часто загружается). Это свойство можно использовать: всегда держать первые блоки на GPU, не выгружать их в CPU.
