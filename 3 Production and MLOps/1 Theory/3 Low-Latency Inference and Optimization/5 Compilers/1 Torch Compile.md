# torch.compile: JIT-компиляция моделей в PyTorch 2.x

## 1. Определение и мотивация

### 1.1 Проблема eager execution

PyTorch по умолчанию работает в режиме eager execution: каждая операция (torch.add, torch.matmul) выполняется «как написана». Это даёт гибкость (можно дебажить, менять граф на лету, использовать print), но неэффективно:

- **Launch overhead.** Каждая операция запускает отдельный CUDA-кернел. Для модели с 1000 операций overhead на запуск: 1000 × 5–20 мкс = 5–20 мс на шаг.
- **Нет fusion.** Соседние операции читают и пишут в HBM десятки раз, хотя можно выполнить всё за один проход.
- **Нет autotuning.** Размер блока и количество регистров фиксированы — не адаптируются под конкретную GPU.

torch.compile решает эти проблемы: перехватывает граф вычислений, оптимизирует его и кеширует оптимизированную версию для повторного использования.

### 1.2 Как это выглядит

```python
import torch

model = MyModel().cuda()

# Без компиляции (eager) — гибко, но медленно
y = model(x)

# С компиляцией
model_compiled = torch.compile(model)
y = model_compiled(x)  # первый вызов: capture + compile (медленно)
y = model_compiled(x)  # второй вызов: оптимизированный кернел (быстро)
```

**Первый вызов** — компиляция. Сборка графа, оптимизация, генерация Triton-кернелов. В 10–100 раз медленнее eager.

**Второй вызов** — выполнение сгенерированных кернелов. В 1.5–5 раз быстрее eager, в зависимости от модели и режима.

## 2. Архитектура

torch.compile состоит из трёх слоёв:

```
PyTorch model (eager)
       │
       ▼
TorchDynamo (bytecode capture)
       │
       ▼
TorchInductor / Backend (code generation)
       │
       ▼
Triton kernels / C++ code
```

### 2.1 TorchDynamo — перехват графа

**TorchDynamo** перехватывает Python-байткод на лету. Он анализирует фреймы стека CPython при выполнении и извлекает фрагменты графа, которые не содержат контроля потока (if, for, while). Эти фрагменты называются **графлетами** (graphlets).

```python
def forward(self, x, mask):
    x = self.linear(x)            # графлет 1 (линейный)
    x = torch.where(mask, x, 0)  # TorchDynamo останавливает граф здесь
    x = self.dropout(x)           # графлет 2
    return x
```

Если граф динамический (содержит `if` по данным), Dynamo разбивает его на несколько графлетов — каждый компилируется отдельно. Overhead от переключения между графлетами невелик, если их мало.

**Когда Dynamo не может перехватить граф:**
- Вызовы внешних C-библиотек (например, кастомные CUDA-расширения).
- Использование `torch.jit.script` или `torch.jit.trace` внутри модели.
- Некоторые операции из `torch.*`, не поддерживаемые Dynamo (редко).

### 2.2 TorchInductor — генерация кода

**TorchInductor** — бэкенд по умолчанию. Получает графлет от Dynamo и генерирует:

- **Triton-кернелы** для NVIDIA GPU (через язык Triton, не путать с NVIDIA Triton Inference Server).
- **C++ код** с OpenMP для CPU.

TorchInductor применяет:

1. **Fusion (слияние операций).** Несколько последовательных операций объединяются в один кернел:

   ```
   До: matmul → add → relu  (3 кернела, 3 чтения/записи в HBM)
   После: fused_matmul_add_relu (1 кернел, 1 запись в HBM)
   ```

2. **Tiling (разбиение на блоки).** Большие тензоры разбиваются на блоки, которые помещаются в shared memory — меньше обращений к HBM.

3. **Autotuning.** Для каждой операции Inductor перебирает десятки вариантов кернела (разные размеры блока, количество регистров, unrolling факторы) на этапе компиляции и выбирает самый быстрый.

## 3. Режимы компиляции

```python
# Компромисс скорость компиляции / скорость выполнения
model = torch.compile(model, mode="default")

# Максимальная оптимизация (долгая компиляция)
model = torch.compile(model, mode="max-autotune")

# Быстрый старт (минимальные оптимизации)
model = torch.compile(model, mode="reduce-overhead")
```

| Режим | Compile time | Performance | Когда использовать |
|-------|-------------|-------------|-------------------|
| `default` | Секунды | 1.5–3× | Большинство случаев, баланс |
| `max-autotune` | Минуты (до 30 для 70B) | 2–5× | Production, когда можно долго ждать первый запрос |
| `reduce-overhead` | <1 сек | 1.2–2× | Разработка, частые изменения модели |

### 3.1 max-autotune

В режиме `max-autotune` Inductor перебирает все возможные варианты кернелов для каждой операции. Для LLM это может занять 10–30 минут. Результат — максимально быстрый исполняемый код. Кешируется — повторные запуски не перекомпилируют.

```python
# После компиляции кеш сохраняется в:
# /tmp/torchinductor_<username>/
# При повторном запуске той же модели компиляция не нужна
```

### 3.2 fullgraph

Если модель — один статический граф (нет if/for по данным), можно запретить Dynamo разбивать его. Это даёт более агрессивные оптимизации:

```python
model = torch.compile(model, fullgraph=True)
```

Если граф динамический — будет ошибка. Полезно для production, когда уверены в статичности модели.

## 4. Опции и конфигурация

### 4.1 Backend

По умолчанию — `inductor`. Можно указать другой:

```python
# Только для отладки: eager — без компиляции
model = torch.compile(model, backend="eager")

# AOTAutograd — компиляция под обучение (autograd-aware)
model = torch.compile(model, backend="aot_autograd")

# NVFuser (PyTorch < 2.0, устарел)
model = torch.compile(model, backend="nvfuser")
```

### 4.2 dynamic=True

По умолчанию torch.compile оптимизирует под те размеры тензоров, которые были на первом вызове. Если размер меняется, компиляция происходит заново.

```python
# Включить поддержку динамических размеров
model = torch.compile(model, dynamic=True)

# Это жертвует производительностью ради гибкости
# Размеры тензоров кодируются как параметры, а не константы
```

### 4.3 Environment variables

```bash
# Отладка: показать сгенерированный код
TORCHINDUCTOR_MAX_AUTOTUNE=1 TORCH_COMPILE_DEBUG=1 python train.py

# Выключить компиляцию глобально
TORCHDYNAMO_DISABLE=1 python train.py

# Указать директорию кеша
TORCHINDUCTOR_CACHE_DIR=/path/to/cache python train.py
```

## 5. Границы применимости

### 5.1 Когда torch.compile эффективен

- **Статические графы.** Модель не меняет структуру от вызова к вызову (большинство LLM).
- **GPU-инференс.** Triton-кернелы дают наибольший выигрыш на NVIDIA GPU.
- **Большие тензоры.** Для маленьких тензоров (batch=1, seqlen=1) overhead от компиляции может превышать выигрыш.
- **Повторяющиеся вызовы.** Первый вызов — компиляция, остальные — ускорение.

### 5.2 Когда неэффективен

- **CPU.** Inductor генерирует C++ с OpenMP, но выигрыш меньше, чем на GPU (1.2–1.5×).
- **Одиночные вызовы.** Если модель вызывается один раз (batch job), compile time не окупается.
- **Модели с кастомными CUDA-расширениями.** Dynamo не может перехватить граф через кастомный код.
- **Очень маленькие модели (< 100M параметров).** Launch overhead и так мал, выигрыш от fusion незначительный.

### 5.3 Compile time для LLM

| Модель | Режим | Compile time |
|--------|-------|-------------|
| ResNet-50 | default | 2–3 сек |
| ResNet-50 | max-autotune | 60–120 сек |
| LLaMA-7B | default | 10–20 сек |
| LLaMA-7B | max-autotune | 5–10 мин |
| LLaMA-70B | default | 2–5 мин |
| LLaMA-70B | max-autotune | 20–40 мин |

**Прогрев (warmup):** Запустить модель на синтетических данных до начала обслуживания запросов. Тогда первый реальный запрос получит оптимизированный кернел.

## 6. torch.compile vs torch.jit (старый подход)

| | torch.jit.trace | torch.jit.script | torch.compile |
|---|---|---|---|
| Метод | Трассировка | Анализ AST | Bytecode capture |
| Поддержка динамики | Нет (trace фиксирует граф) | Частично (if/for) | Автоматически (graphlets) |
| Ошибки | Тихие (trace видит только выполненные пути) | Громкие (не все Python поддерживается) | Минимальные (fallback к eager) |
| Поддержка | Устарел | Устарел | Актуальный |
| Ускорение | 1–1.5× | 1–2× | 1.5–5× |

Основное преимущество torch.compile — **безопасность**. Если Dynamo не может скомпилировать фрагмент, он оставляет его в eager режиме. Модель не ломается — просто работает медленнее на этом участке.

## 7. Типичные ошибки

- **Забыть прогрев.** Первый вызов — компиляция (10–30 сек). Если это production-запрос, latency будет неприемлема. Всегда делать warmup при старте.
- **max-autotune для экспериментальной модели.** Compile time в минуты для модели, которую вы меняете каждые 10 минут — пустая трата времени.
- **dynamic=True без необходимости.** Жертвует 10–20% производительности в пользу гибкости. Если размеры входов стабильны — не нужен.
- **Компиляция маленьких тензоров.** Для batch=1 и seqlen=1 overhead от Triton-кернелов может превысить выигрыш.
- **Не кешировать.** Удаление /tmp/torchinductor_* каждый раз — compile time заново.
