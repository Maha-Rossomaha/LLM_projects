# CI/CD Intro

## 1. Что такое CI/CD

**CI/CD** — практика автоматизации сборки, тестирования и доставки кода от разработчика до продакшена.

**Проблема без CI/CD:**
* Разработчик собирает и деплоит вручную → ошибки, забытые шаги,
* Тесты запускаются выборочно → регрессии не замечены,
* Деплой — «ритуал» по инструкции из wiki → неповторяемо,
* Версии моделей — `model_v3_final_fixed.pkl` → хаос.

**Что даёт CI/CD:**
* Каждое изменение кода автоматически собирается и тестируется (CI),
* Прошедший проверки артефакт автоматически или по кнопке доставляется в окружение (CD),
* Весь процесс описан в коде (Pipeline as Code) и версионируется в Git,
* Откат на любую предыдущую версию — одна команда.

## 2. CI — Continuous Integration

**Continuous Integration** — практика частой (несколько раз в день) интеграции изменений в общую ветку с автоматической сборкой и тестированием.

**Что делает CI:**
1. Разработчик пушит код в репозиторий или создаёт Merge Request.
2. Автоматически запускается pipeline:
   * **Lint** — проверка стиля кода (flake8, ruff, eslint).
   * **Unit-тесты** — быстрые тесты отдельных функций/модулей.
   * **Сборка** — компиляция, сборка Docker-образа.
   * **Интеграционные тесты** — проверка взаимодействия модулей.
   * **Security scanning** — SAST, SCA, поиск секретов.
3. Если все проверки пройдены — изменения можно вливать в основную ветку.

**Ключевой принцип CI:** изменения должны проверяться **как можно раньше** и **автоматически**. Чем дольше ветка живёт без интеграции — тем больнее слияние.

**Пример пайплайна CI:**
```
Push в ветку → Lint → Unit-test → Build Docker image → Integration test → Security scan → Готов к деплою
```

## 3. CD — Continuous Delivery / Continuous Deployment

**Continuous Delivery (CD)** — практика, при которой каждое прошедшее CI изменение **готово к деплою** в прод, но сам деплой выполняется вручную (по кнопке).

**Continuous Deployment** — более строгий вариант: каждое прошедшее CI изменение **автоматически деплоится** в прод без ручного вмешательства.

**Разница:**
| | Continuous Delivery | Continuous Deployment |
|---|---|---|
| Деплой в прод | Ручной (manual approval) | Автоматический |
| Когда применять | Критичные сервисы, регуляторные требования | Высокая зрелость процессов, доверие к тестам |
| Риск | Ниже (человек принимает решение) | Выше (автоматика, нужны жёсткие quality gates) |

**Что делает CD:**
1. Берёт артефакт, прошедший CI (Docker-образ, пакет, модель).
2. Деплоит в staging-окружение.
3. Запускает post-deploy проверки (smoke tests, performance tests).
4. При успехе — деплоит в production (автоматически или после manual approval).
5. При провале — автоматический откат.

**Пример пайплайна CD:**
```
CI пройден → Deploy staging → E2E test → Perf test → [Manual approval] → Deploy prod → Post-deploy check
```

## 4. CI/CD Pipeline — общая картина

```
                  CI                                      CD
┌──────────────────────────────────────┐ ┌─────────────────────────────────────────┐
│                                      │ │                                         │
│  Push/MR → Lint → Unit → Build →     │ │  Deploy   →  Smoke → E2E →  Perf →      │
│            SAST → SCA → Integration  │ │  staging     test    test   test        │
|                                      | |                                         |
│                                      │ │                ▼                        │
│              ▲ gate: все тесты       │ │         gate: quality metrics           │
│                 зелёные              │ │          в пределах SLA                 │
│                                      │ │                                         │
└──────────────────────────────────────┘ │                 ▼                       │
                                         │   [Manual approval ─── or ─── auto]     │
                                         │                 ▼                       │
                                         │   Deploy prod → Post-deploy check       │
                                         │                                         │
                                         └─────────────────────────────────────────┘
```

**Gate** — «ворота», условие для прохождения на следующий этап. Если gate не пройден — pipeline останавливается, артефакт не идёт дальше.

## 5. Pipeline as Code

Вся логика CI/CD описывается **в коде** и хранится в том же Git-репозитории, что и код приложения:

* **GitLab CI** — `.gitlab-ci.yml`
* **GitHub Actions** — `.github/workflows/*.yml`
* **Jenkins** — `Jenkinsfile`
* **ArgoCD** — `Application CRD`

**Почему это важно:**
* Pipeline версионируется вместе с кодом,
* Изменения в pipeline проходят code review,
* Воспроизводимость: pipeline для старого тега запустится точно так же,
* Аудит: кто, когда и что изменил в процессе деплоя.

## 6. Связь с ML-контекстом

Для ML-сервисов CI/CD имеет особенности:

* **Артефакт CI** — не только Docker-образ, но и модель (веса, конфиг, токенизатор).
* **Quality gates** — не только unit/e2e, но и метрики модели (MRR, accuracy, drift).
* **Model Registry** — центральное звено между обучением и деплоем.
* **Staging** — не просто «тестовый кластер», а окружение с реальными данными для валидации модели.
* **Откат** — не только кода, но и модели (предыдущая версия в Model Registry).

**Типовой ML CI/CD flow:**
```
Train model → Register in Model Registry → CI: build + test → CD: staging → Validate model → Prod
```

## 7. Чек-лист

1. **CI**: pipeline запускается автоматически на push/MR, включает lint, unit, build, security.
2. **CD**: staging-деплой автоматически, prod — автоматически или manual approval.
3. **Pipeline as Code**: конфиг CI/CD в Git, версионируется.
4. **Gates**: формальные условия перехода между этапами.
5. **Rollback**: есть процедура возврата на предыдущую версию.
6. **Для ML**: Model Registry и quality-метрики модели встроены в pipeline.

## 8. Типичные ошибки

1. **CI и CD — одно и то же**: CI — сборка и тестирование, CD — доставка до окружения. Разные этапы, разные задачи.
2. **Continuous Delivery ≠ Continuous Deployment**: Delivery требует ручного подтверждения перед продом, Deployment — нет.
3. **Pipeline без gates**: любой зелёный pipeline идёт в прод → ошибка в тестах, но модель уже деплоится.
4. **Деплой в прод без staging**: нет проверки на реальном окружении → инцидент.
5. **Ручной деплой вне pipeline**: обход CI/CD → нет аудита, нет воспроизводимости.
