# DNS: как работает резолвинг и как диагностировать

## 1. Что такое DNS и что он делает

### 1.1. Базовая задача
DNS (Domain Name System) превращает имя хоста в данные, которые нужны клиенту.
Чаще всего это:
- имя → IP-адрес (чтобы потом подключиться по TCP/UDP),
но DNS может хранить и другие типы данных (почта, сервисы, ключи и т.д.).

Пример цепочки в жизни:
1) приложение знает имя `api.example.com`
2) делает DNS-запрос и получает IP
3) затем уже делает TCP соединение на `IP:порт` (например, 443)

### 1.2. Важно не путать слои
- DNS отвечает на вопрос: “какой IP (или другие данные) у этого имени?”
- TCP/firewall отвечает на вопрос: “доступен ли сервис на этом IP:порт?”

Поэтому “сервис не работает” почти всегда начинается с проверки DNS.

## 2. Кто участвует в DNS

### 2.1. Клиент (stub resolver)
Это часть ОС/библиотек, которая:
- берёт имя (`example.com`)
- отправляет запрос на DNS-сервер (обычно тот, что указан в настройках системы)

### 2.2. Рекурсивный резолвер (recursive resolver)
Это DNS-сервер, который делает работу “под ключ”:
- принимает твой запрос
- сам ходит по цепочке DNS-серверов (если нужно)
- собирает ответ и возвращает его тебе
- кэширует результат на время TTL

Примеры рекурсивных резолверов:
- DNS провайдера
- публичные резолверы (1.1.1.1, 8.8.8.8)
- CoreDNS в Kubernetes (для сервисных имён)

### 2.3. Authoritative DNS (авторитетный сервер)
Это сервер, который “владеет правдой” для конкретной зоны (например, `example.com`):
- он не делает рекурсию “по всему интернету”
- он отвечает авторитетно: какие записи есть у домена и какие у них TTL

У домена обычно несколько authoritative-серверов (NS), чтобы была отказоустойчивость.

## 3. Типы DNS-записей (что означают)

### 3.1. A и AAAA
- `A` — имя → IPv4
- `AAAA` — имя → IPv6

Пример: `api.example.com A 93.184.216.34`

### 3.2. CNAME
`CNAME` — “это имя является алиасом на другое имя”.
Пример:
- `www.example.com` CNAME `example.com`
Дальше резолвер отдельно резолвит `example.com` в A/AAAA.

Практический смысл:
- удобно менять “цель” в одном месте
- но добавляет шаг в резолвинге (ещё одно имя → ещё один запрос/кэш)

### 3.3. MX
`MX` — почтовые серверы домена (куда доставлять почту для `@example.com`).
Обычно не нужно для сервисов, но встречается в инфраструктуре.

### 3.4. TXT
`TXT` — произвольные текстовые данные.
Частые использования:
- SPF/DKIM/DMARC в почте
- доменные верификации (например, для TLS/сертификатов, сервисов)
- иногда конфигурации для сервисов

### 3.5. NS
`NS` — какие authoritative-сервера отвечают за зону.
Это основа делегирования в DNS.

### 3.6. SRV
`SRV` — “где находится сервис” (хост + порт + приоритет/вес).
Формат имени обычно такой:
`_<service>._<proto>.<name>`

Примеры:
- `_sip._tcp.example.com`
- `_grpc._tcp.service.namespace.svc.cluster.local` (внутри k8s могут быть похожие схемы)

Практический смысл:
- сервис-дискавери: клиент узнаёт не только хост, но и порт, и может выбирать из нескольких целей.

## 4. TTL и кэширование

### 4.1. Что такое TTL
TTL (Time To Live) — “сколько секунд можно кэшировать ответ”.
TTL задаётся в записи authoritative-сервером.

### 4.2. Где кэшируется
Кэш может быть в нескольких местах:
- в приложении/библиотеке (иногда)
- в ОС (иногда)
- в рекурсивном резолвере (почти всегда)

### 4.3. Почему “DNS не применился сразу”
Если ты поменял запись, но у кого-то остался кэш на старый ответ, то до истечения TTL он может продолжать видеть старое значение.

Практический вывод:
- после изменений DNS надо помнить про TTL
- “у меня уже работает, у коллеги ещё нет” часто объясняется разными кэшами

## 5. search domain и ndots (важно для Kubernetes)

### 5.1. Search domain (суффиксы поиска)
Если ты запрашиваешь имя без точки, например `api`, система может автоматически пробовать:
- `api.<search-domain-1>`
- `api.<search-domain-2>`
- …

Search domains берутся из настроек резолвера (например, `/etc/resolv.conf`).

Пример: если search domain = `corp.local`, то запрос `api` может стать `api.corp.local`.

### 5.2. ndots (порог “считать ли имя полным”)
`ndots` — настройка, сколько точек в имени нужно, чтобы считать его “полным” (FQDN) и сначала пробовать “как есть”.

- Если точек меньше `ndots`, система часто сначала пробует добавить search domains.
- Если точек достаточно, сначала пробует резолвить как FQDN.

Почему это болит в k8s:
- там много search domains вида `svc.namespace.svc.cluster.local`
- и короткие имена могут резолвиться через несколько попыток, что добавляет задержку
- иногда приводит к неожиданным запросам “не туда” при коллизиях имён

Практический вывод:
- если видишь странные задержки на DNS и много попыток, смотри `search` и `ndots` в `/etc/resolv.conf`

## 6. Практика (dig)

### 6.1. Базовые запросы
```bash
dig example.com A
dig example.com AAAA
dig example.com CNAME
```

### 6.2 Проверить конкретный резолвер
```bash
dig @1.1.1.1 example.com
```
**Смысл:** понять, проблема в твоём локальном резолвере/сети или в самом домене.

### 6.3 Посмотреть трассу резолвинга
```bash
dig +trace example.com
```
**Смысл:** увидеть цепочку “корневые → TLD → authoritative” и локализовать, на каком шаге проблема.

### 6.4 SRV записи
```bash
dig _sip._tcp.example.com SRV
```
Что смотреть:

- target (куда идти)
- port (на какой порт)
- priority/weight (как выбирать между несколькими)

## 7. Как отличить “проблема DNS” от “проблема TCP/firewall”
### 7.1. Симптомы DNS-проблем

* `dig name` не возвращает A/AAAA (NXDOMAIN/timeout/servfail)
* имя резолвится в “не тот” IP (не тот адрес/устаревший кэш)

**Порядок действий:**

1. `dig name`
2. `dig @1.1.1.1 name` (или другой внешний резолвер)
3. `dig +trace name` (если нужно понять, где ломается)

### 7.2. Симптомы TCP/firewall-проблем

* DNS даёт IP, но подключение не работает:
  * `timeout` (дроп/нет доступа/маршрут/правило)
  * `refused` (никто не слушает порт)
  * ошибки TLS/HTTP выше по стеку

Порядок действий:

1. убедиться, что DNS даёт IP
2. проверить доступность порта: `curl -v http://IP:port` или `curl -v https://name`
3. при необходимости смотреть состояния соединений: `ss -tanp`
