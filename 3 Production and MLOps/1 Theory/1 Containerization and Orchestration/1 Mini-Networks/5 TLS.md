# TLS: шифрование, сертификаты, SNI — конспект

## 1. Что такое TLS и что он даёт

TLS (Transport Layer Security) — это слой поверх TCP, который делает соединение защищённым.

1.1. TLS обычно даёт три свойства:

1.1.1. Конфиденциальность  
Данные шифруются, посторонний на пути не может прочитать содержимое.

1.1.2. Целостность  
Данные защищены от незаметной подмены: если кто-то изменит байты “в дороге”, это будет обнаружено.

1.1.3. Аутентификация сервера (обычный сценарий HTTPS)  
Клиент проверяет, что он подключился именно к тому серверу/домену, который ожидал, по сертификату.

Замечание: TLS не “делает приложение безопасным”, он защищает канал. Ошибки логики/доступа уровня приложения TLS не исправляет.

## 2. Сертификат: что это и зачем

2.1. Сертификат сервера — это документ (X.509), где указано:
- для каких имён (доменов) он действителен;
- кто его выпустил (какой центр сертификации, CA);
- срок действия;
- публичный ключ сервера (и служебные поля).

2.2. Зачем он нужен
Клиенту нужно понять: “я подключился к тому, кому доверяю”.
Клиент доверяет не “любому сертификату”, а цепочке доверия до известного CA.

## 3. Цепочка доверия: leaf → intermediate → root

3.1. Идея цепочки
Обычно сервер не ограничивается одним сертификатом. У него есть цепочка:

3.1.1. Leaf (сертификат сервера)  
Выдан на конкретное имя/имена (например, `example.com`).

3.1.2. Intermediate (промежуточный CA)  
Подписывает leaf. Таких уровней может быть несколько.

3.1.3. Root (корневой CA)  
Это “точка доверия”, которая заранее есть у клиента в хранилище доверенных корневых сертификатов (trust store).

3.2. Что проверяет клиент
- подписи по цепочке: leaf подписан intermediate, intermediate подписан root;
- что цепочка приводит к root, которому клиент доверяет;
- что сертификаты не истекли и не “будущие”;
- что сертификат подходит под имя хоста (см. SAN);
- дополнительные проверки (например, политики, иногда ревокация).

Практическая мысль: ошибка “unknown authority” почти всегда означает “цепочка не приводит к доверенному root”.

## 4. CN vs SAN: почему CN уже не главное

4.1. Где хранится имя хоста
Исторически имя хоста писали в поле **CN** (Common Name).

4.2. Современное правило
Сегодня имя/имена, для которых сертификат действителен, должны быть в **SAN** (Subject Alternative Name).
- SAN может содержать несколько DNS-имён (например, `example.com`, `www.example.com`).
- Многие клиенты уже не считают CN достаточным, если SAN отсутствует.

Практическая мысль: “host mismatch” чаще всего значит, что запрошенное имя не входит в SAN.

## 5. SNI: зачем оно нужно

5.1. Проблема одного IP и многих сайтов
На одном IP (или за одним балансировщиком) может обслуживаться много доменов.
Сервер должен понять, какой сертификат показать.

5.2. Что делает SNI
SNI (Server Name Indication) — это расширение TLS, где клиент сообщает серверу имя хоста прямо в начале рукопожатия.
- Клиент говорит: “я иду к `example.com`”.
- Сервер выбирает правильный сертификат именно для этого домена.

5.3. Что бывает без SNI
Если клиент не отправляет SNI (или отправляет другое имя), сервер может выдать “сертификат по умолчанию”.
Тогда клиент видит “не тот сертификат” и получает ошибку несоответствия имени.

## 6. Handshake (очень грубо, по шагам)

Цель handshake: договориться о параметрах шифрования и убедиться в личности сервера.

6.1. ClientHello  
Клиент отправляет:
- поддерживаемые версии TLS и наборы шифров;
- случайные данные (для криптографии);
- SNI (если используется);
- дополнительные расширения.

6.2. ServerHello  
Сервер выбирает параметры:
- версию TLS;
- набор шифров;
- отправляет свои данные для согласования ключей.

6.3. Сертификаты  
Сервер отправляет сертификат и обычно цепочку (leaf + intermediates).

6.4. Проверка сертификата клиентом  
Клиент проверяет цепочку доверия, сроки, имя хоста (SAN), корректность.

6.5. Согласование ключей  
Стороны договариваются о сессионных ключах (ключи для симметричного шифрования).

6.6. Завершение handshake  
После этого начинается защищённая передача данных приложения (например, HTTP).
