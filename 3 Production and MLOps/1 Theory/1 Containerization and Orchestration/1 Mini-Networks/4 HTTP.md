# HTTP поверх TCP

Идея: HTTP — это уровень “что именно просим у сервера”, а TCP — “как байты доезжают”.  
`curl` позволяет одновременно видеть и HTTP-часть (методы, заголовки, коды), и транспортные детали (DNS, TCP, TLS, тайминги).

## 1. Базовая модель: что происходит при запросе

Когда ты делаешь `curl https://site` происходит цепочка:

1) DNS: имя `site` → IP  
2) TCP: соединение к `IP:443`  
3) TLS (если https): рукопожатие, проверка сертификата  
4) HTTP: отправка запроса, получение ответа  
5) (опционально) соединение остаётся открытым для следующих запросов

Тормоза и ошибки обычно можно привязать к одному из этих шагов.

## 2. HTTP запрос: из чего он состоит

### 2.1. Стартовая строка (request line)
Содержит:
- метод: `GET`, `POST`, `PUT`, `DELETE`, …
- путь: `/api/v1/items`
- версия: `HTTP/1.1`

Пример:
`GET / HTTP/1.1`

### 2.2. Заголовки (headers)
Это пары `Key: Value`, которые описывают запрос. Частые:
- `Host`: какой домен ты хочешь (важно, если один IP обслуживает много сайтов)
- `User-Agent`: кто ты
- `Accept`: какие форматы ответа принимаешь
- `Content-Type`: какой формат у тела запроса (часто JSON)
- `Authorization`: аутентификация
- `Cookie`: отправка cookies на сервер

### 2.3. Тело (body)
Есть не всегда.
- у `GET` чаще всего тела нет
- у `POST/PUT` тело обычно есть (например JSON)

## 3. HTTP ответ

### 3.1. Статусная строка (status line)
Пример:
`HTTP/1.1 200 OK`

Где `200` — статус-код.

### 3.2. Заголовки ответа
Частые:
- `Content-Type`: формат ответа (`application/json`, `text/html`, …)
- `Content-Length`: размер тела (если известен)
- `Set-Cookie`: сервер устанавливает cookie
- `Location`: адрес для редиректа (вместе со статусом 301/302/307/308)
- `Cache-Control`: правила кэширования

### 3.3. Тело ответа
Содержимое (HTML, JSON, файл и т.п.).

## 4. Статус-коды: как их читать

### 4.1. 2xx — успех
- `200` — ок
- `201` — создано (часто при POST)

### 4.2. 3xx — редиректы
- `301/302` — “иди на другой URL”
- `307/308` — редирект с сохранением метода (важно для POST)

### 4.3. 4xx — проблема со стороны клиента
- `400` — плохой запрос
- `401` — нужна авторизация
- `403` — доступ запрещён
- `404` — не найдено
- `429` — слишком много запросов (rate limit)

### 4.4. 5xx — проблема со стороны сервера
- `500` — внутренняя ошибка
- `502` — плохой ответ от апстрима (часто прокси/балансировщик)
- `503` — сервис недоступен (перегруз/выключен/нет здоровых инстансов)
- `504` — таймаут на прокси/балансировщике

Практическая мысль: если ты видишь 4xx/5xx — TCP обычно уже работает, проблема выше по стеку.

## 5. keep-alive и повторное использование соединений

### 5.1. Что происходит без keep-alive
Если каждое действие открывает новое TCP-соединение:
- лишний TCP handshake
- для HTTPS ещё и TLS handshake
- больше задержка и нагрузка

### 5.2. HTTP keep-alive
HTTP keep-alive — это практика “не закрывать TCP после одного запроса/ответа”.
- клиент может отправить следующий запрос по тому же соединению
- это обычно делает библиотека/клиент автоматически

### 5.3. Что важно для диагностики
Иногда проблемы проявляются так:
- “первый запрос после простоя странный, второй норм”
Это часто связано с тем, что соединение долго было idle и где-то на пути его забыли.

## 6. Редиректы

Редирект — это когда сервер отвечает статусом `3xx` и заголовком `Location`.
- браузер или клиент должен сделать новый запрос на другой URL

Типичные случаи:
- `http://site` → `https://site`
- `/` → `/home`
- смена домена

`curl` по умолчанию редиректы не знает. Их надо явно включить.

## 7. Cookies

Cookie — это небольшой кусок данных, который:
- сервер может попросить клиента хранить (`Set-Cookie`)
- клиент отправляет обратно (`Cookie`)

Где используется:
- сессии
- авторизация
- персонализация
- A/B-тесты

Для диагностики важно:
- видеть `Set-Cookie` в ответе
- понимать, что поведение может зависеть от cookies

## 8. HTTP/1.1 vs HTTP/2 
    
### 8.1. HTTP/1.1 (эффекты)
- запросы/ответы идут “последовательно” внутри одного соединения
- для параллелизма часто открывают несколько соединений
- заголовки не оптимизированы по размеру

### 8.2. HTTP/2 (эффекты)
- несколько потоков внутри одного TCP-соединения (мультиплексирование)
- обычно меньше потребность в множестве соединений
- заголовки сжимаются (HPACK)
- часто быстрее на множестве мелких запросов

Важная практическая деталь:
- и HTTP/1.1, и HTTP/2 обычно работают поверх TCP
- проблемы TCP (потери/таймауты) могут влиять на оба, но ощущаться по-разному

## 9. Практика: команды curl и что в них смотреть

### 9.1. Показать детали запроса/ответа
```bash
curl -v http://site
```

Что смотреть в выводе:

* на какой IP подключился (после DNS)
* какая версия HTTP
* какие заголовки отправил/получил
* какой статус-код пришёл

### 9.2. Только заголовки ответа

```bash
curl -I http://site
```

Полезно, чтобы быстро увидеть:

* статус-код
* Location (если редирект)
* Set-Cookie
* Content-Type, Cache-Control

### 9.3 Редиректы (следовать Location)
```bash
curl -vL http://site
```

`-L` делает так, что `curl` повторяет запросы по цепочке редиректов.
Что смотреть:
* сколько редиректов
* куда именно перекидывает
* не попал ли ты в цикл

### 9.4. Тайминги диагностики
```bash
curl -w "\nDNS:%{time_namelookup} TCP:%{time_connect} TLS:%{time_appconnect} TTFB:%{time_starttransfer} TOTAL:%{time_total}\n" -o /dev/null -s https://site
```

Что означают поля:
* `time_namelookup` — сколько занял DNS (имя → IP)
* `time_connect` — сколько заняло установление TCP
* `time_appconnect` — сколько занял TLS (для https)
* `time_starttransfer` (TTFB) — сколько времени до первого байта ответа (DNS+TCP+TLS + работа сервера до начала ответа)
* `time_total` — всё целиком (включая скачивание тела)
