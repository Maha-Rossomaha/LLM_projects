# Networks Intro 

## 0. Карта уровней 

**Сеть/транспорт**: IP, маршрутизация, TCP/UDP, порты  
**Имена**: DNS (как имя превращается в IP)  
**Шифрование**: TLS/mTLS (как HTTP становится HTTPS)  
**Приложение**: HTTP, gRPC (что реально “говорят” сервисы)

**Ключевая мысль:** почти любой кейс “сервис недоступен” раскладывается по слоям. Ты идёшь снизу вверх и на каждом слое задаёшь простой вопрос: *это работает?*.

## 1. IP и подсети

### 1.1 Теория

1. **Host** — любой узел в сети, который участвует в обмене по сети и может отправлять/принимать пакеты.

2. **Сеть** — это группа хостов, между которыми есть путь передачи данных по общим правилам (IP).

3. **IP-адрес** — это идентификатор хоста (точнее: его сетевого интерфейса) в этой системе доставки пакетов.
    - IPv4: `192.168.1.10` (4 числа 0–255)
    - IPv6: длинный адрес вроде `2001:db8::1` (реже в локальных лабах, но в проде встречается)

4. **Подсеть / CIDR** — правило “какие адреса считаются рядом/в одной сети”.
    - Пример: `192.168.1.0/24`
    - `/24` значит: первые 24 бита (то есть первые 3 числа) — это номер сети, оставшиеся — номер хоста внутри этой сети.
    - В /24 обычно ~256 адресов (на практике часть зарезервирована).

5. **Приватные диапазоны (IPv4)** — специальные диапазоны IPv4, которые предназначены для внутренних сетей: дома, в офисе, в датацентре, внутри Docker/K8s.  
   **Идея:** внутри своей сети можно раздавать адреса как угодно (например, всем устройствам 192.168.1.*), и это никому “снаружи” не мешает.
   #### Мини-объяснение 
   Маршрутизатор в интернете (у провайдера и дальше) не будет пересылать пакеты, у которых:
   * адрес назначения приватный (10.x, 172.16–31.x, 192.168.x), или
   * часто и адрес источника приватный
   ```
   То есть если из дома попробовать отправить пакет напрямую на 10.1.2.3, интернет “скажет”:
   я не знаю, где это, и не обязан знать, потому что 10.* — это чей-то внутренний адрес.
   ```  
   Поэтому приватные адреса живут только внутри локальных сетей. **Как тогда работает интернет из дома:** 
   1. устройства внутри дома получают приватные адреса (например, `192.168.1.5`)
   2. роутер на границе делает NAT: “переводит” внутренние адреса в один публичный адрес роутера, который уже виден в интернете 

   **Пример:**
   * ноут: `192.168.1.5` (приватный)
   * роутер снаружи: `93.184.216.34` (публичный)
   * когда ноут идёт на сайт, роутер подменяет “от кого” — и сайт отвечает на публичный адрес роутера, а роутер уже возвращает ответ ноуту

   #### Почему Docker/K8s часто показывают `10.*` или `172.*`  
   Потому что в них строятся внутренние сети:
   * Docker создаёт приватную подсеть для контейнеров
   * Kubernetes создаёт приватные подсети для подов/сервисов  

   Эти адреса нужны, чтобы процессы могли общаться внутри инфраструктуры, а наружу обычно выходит уже через NAT/балансировщики/ingress.

6. **Интерфейс** — это “сетевой вход/выход” в системе. Через интерфейсы ОС отправляет и получает сетевые пакеты. У одного хоста может быть несколько интерфейсов, и IP-адрес обычно назначается именно интерфейсу.
    - **`lo` (loopback)**
        - Это виртуальный интерфейс “сам с собой”.
        - Адрес `127.0.0.1` (и имя `localhost`) означает: подключись к сервису на этой же машине, не выходя в сеть вообще.
        - Если сервер слушает только на `127.0.0.1`, то он доступен только локально.
    - **`eth0` (или похожие)** 
        - Это “обычный” интерфейс, через который машина общается с внешним миром: с роутером, с интернетом, с другими машинами.
        - На нём висит “внешний” для этой машины IP (например, `192.168.1.10` дома, или `10.x` внутри облака).
        - Если сервер слушает на этом интерфейсе (или на `0.0.0.0`), к нему могут прийти другие машины.
    - **`veth*`, `br-*` и т.п.**
        - Это интерфейсы, которые появляются, когда система делает “виртуальную сеть”:
          - **br-\* (bridge)** — как виртуальный “свитч” внутри твоей машины: к нему подключают виртуальные штуки.
          - **veth\*** — “виртуальный провод” (пара интерфейсов), который соединяет, например, контейнер с этим bridge.

## 2. Маршрутизация и default gateway

### Теория

1. **Маршрут** — это запись вида:
    - для какой сети/диапазона адресов (например 172.27.208.0/20)
    - через какой интерфейс отправлять (например dev eth0)
    - иногда ещё через какой следующий хоп (шлюз, via 172.27.208.1)  
  
    **Смысл:** “если адрес назначения попадает в этот диапазон — отправляй туда”.
    
    **Пример логики:** если ты шлёшь на `172.27.211.50`, и у тебя есть маршрут `172.27.208.0/20 dev eth0` — ОС понимает: это “рядом”, можно отправлять через `eth0` в локальный сегмент.

2. **Default route** — это “маршрут на всё остальное”, обычно выглядит как:
    - `default via <gateway_ip> dev eth0`

    **Смысл:** если адрес назначения не попал ни в один более конкретный маршрут, отправь пакет на шлюз.  

    **Шлюз (gateway)** — это устройство/узел, который “умеет дальше”: он либо пересылает в другие сети, либо в интернет.  

    **Почему default важен:**
      - без default ты сможешь общаться только с “локальными” сетями, про которые есть конкретные маршруты но не сможешь ходить “куда угодно” (например, на публичные IP сайтов) 
      - если адрес назначения не попал ни в один конкретный маршрут → идём в default.

3. **Как ОС выбирает маршрут**  
  Когда ты подключаешься к IP, ОС:
    - берёт адрес назначения
    - пробегается по таблице маршрутов
    - выбирает самый специфичный маршрут (самая длинная маска, /24 предпочтительнее /16)
    - если ничего не подошло — берёт default
    
    Это ключевая идея: “самое конкретное правило выигрывает”

## 3. Порт, сокет, соединение

### Теория

1. **Порт** — номер (0–65535), который ОС использует, чтобы понять какому процессу отдать входящее соединение/пакет.  
  Как это работает в реальности:
    - На один IP может прийти много разных “типов трафика”. Чтобы различать, к кому он относится, используют порт.
    - Серверный процесс “занимает” порт:
      - он открывает сокет и делает `listen` на, например, `0.0.0.0:8080`.
      - С этого момента ОС знает: “всё, что приходит на порт `8080` — отдавать этому процессу”.
    - Клиент подключается не просто к IP, а к IP:порт: `172.27.211.226:8080`.
    - Примеры по умолчанию: HTTP `80`, HTTPS `443`, SSH `22` но это условность:
      - это просто общепринятые значения по умолчанию, чтобы людям и софту было удобно.
      - но технически ничто не мешает сделать HTTP на `8080`, `5000`, `12345` — и это часто делают (особенно для внутренних сервисов).
      
    Ещё полезная деталь:
    - сервер обычно слушает на “известном” порту (`80`/`443`/`8080`),
    - а клиент при подключении получает временный (ephemeral) порт у себя, типа `53124`. Поэтому соединение выглядит так: client_ip:53124 -> server_ip:443.

1. **Сокет** — это “ручка/объект” в ОС, через который программа общается по сети (или вообще по вводу-выводу).

    Программа не “шлёт пакеты сама” — она работает с сокетом, а ОС уже делает всю сетевую работу.

    Если говорить только про сетевые сокеты, то сокет можно понимать как: `(protocol, local IP, local port)`, а в случае установленного TCP-соединения добавляется ещё и адрес второй стороны: `(protocol, local IP, local port, remote IP, remote port)`.

    **Зачем он нужен:**
      - Чтобы ОС могла сопоставить трафик процессу.
      - Чтобы у процесса было состояние: открыт/закрыт, буферы, очереди, ошибки и т.п.

    **“Сокет = (IP, port, protocol)” — что это значит**
      - **protocol:** какой транспорт используется (пока просто “тип сокета”)
      - **IP:** на каком адресе этот сокет “привязан” (например, только `127.0.0.1` или на всех `0.0.0.0`)
      - **port:** какой номер порта он занимает/использует
    
    **Два ключевых типа сокетов:**
      - **Слушающий (listening) сокет** — у сервера  
      Пример: сервис “слушает” 0.0.0.0:8080.  
      Это означает: “если кто-то придёт на порт 8080 — ОС передаст запрос этому сервису”.
      - **Сокет соединения** — появляется после подключения  
      Когда клиент подключился, у сервера появляется отдельный сокет именно для этого клиента, уже с парой адресов:  
      local (server_ip:8080) <-> remote (client_ip:client_port).

2. **TCP соединение** однозначно определяется **четвёркой (4-tuple)**: `(src_ip, src_port, dst_ip, dst_port)`  
    * `src_ip` — IP клиента (откуда пришли)
    * `src_port` — порт клиента (обычно временный, выбирается ОС)
    * `dst_ip` — IP сервера (куда пришли)
    * `dst_port` — порт сервера (на котором слушает сервис)
  
    **Почему этого достаточно, чтобы отличать соединения**  
    Пусть 1000 клиентов одновременно подключаются к одному серверу `server_ip:443`.  
    У всех будет одинаковое:
      - `dst_ip = server_ip`
      - `dst_port = 443`  
  
    Но у каждого клиента будут разные:
      - `src_ip` (если клиенты разные)
      - и почти всегда разные `src_port` (даже если клиент один и тот же)  

      Именно поэтому ОС на сервере может понять: “это пакет относится к соединению №123”, и отдать его правильному сокету.

    **Зачем нужен `src_port`**  
    Даже если один клиент открывает много соединений к одному серверу `:443`, IP у него один и тот же. Отличать соединения помогает `src_port`.

    Пример (одна и та же машина открыла два соединения):
      - `172.27.211.226:53124 -> 93.184.216.34:443`
      - `172.27.211.226:53125 -> 93.184.216.34:443`

    Это два разных TCP-соединения, потому что разные `src_port`.

    **Важная практическая деталь**  
    Одинаковая четвёрка не может существовать дважды одновременно в одной машине/стеке — это и есть “уникальный идентификатор” соединения.  
    Поэтому иногда при очень частых переподключениях всплывает `TIME_WAIT`: ОС специально держит состояние, чтобы не перепутать старые пакеты со “вновь созданным соединением с той же четвёркой”.

## 4. TCP vs UDP 

### TCP
TCP — **надёжный поток байт** поверх IP.
- TCP = соединение + поток байт
- гарантирует доставку и порядок (если потерялось — попробует доставить снова)
- следствие: приложение читает не “сообщения”, а куски потока (сегодня пришло 3 KB, завтра ещё 2 KB — и это нормально)

> Важно помнить: TCP — это поток, а не “сообщения”. Границы сообщений делает протокол выше (HTTP, gRPC и т.д.).

### UDP
- UDP = без соединения, отправка “пакетами”
- нет гарантии доставки/порядка
- следствие: если нужно надёжно — надёжность делает приложение/протокол выше

Где встречается:
- DNS чаще UDP/53
- QUIC (HTTP/3) поверх UDP

## 5. DNS: имена → IP

### Теория

**DNS** превращает имя (`service.company.com`) в IP.

**Резолвер** — “кто отвечает на DNS-запросы у тебя”.
- обычно это локальный резолвер/настройка системы
- в Kubernetes часто важна связка search domain / ndots

**Типы записей:**
- `A` — имя → IPv4
- `AAAA` — имя → IPv6
- `CNAME` — алиас на другое имя

**TTL и кэширование**
- DNS ответы кэшируются на время TTL
- поэтому “поменяли запись, но не применилось” часто = кэш

Как отличать проблемы:
- “имя не резолвится” = DNS
- “имя резолвится, но порт недоступен” = TCP/firewall/приложение

## 6. HTTP как протокол

### Теория

**HTTP** — протокол запрос/ответ на уровне приложения.
- в HTTP/1.1 он текстовый (видно заголовки)
- обычно поверх TCP

**Состав запроса:**
- метод: `GET/POST/PUT/DELETE/...`
- путь: `/api/v1/...`
- заголовки (headers)
- тело (body) — часто JSON

**Состав ответа:**
- статус-код: `200/301/400/401/403/404/500/503`
- заголовки
- тело

**Keep-alive** — несколько запросов по одному TCP соединению (меньше накладных расходов).

**Редиректы** — ответы `301/302/307/308`, когда сервер отправляет на другой URL.

**API в терминах HTTP**
- endpoint = конкретный URL/путь
- request body = данные запроса (JSON)
- headers = метаданные (auth, content-type и т.д.)

## 7. gRPC

### Теория

gRPC — популярный протокол для сервис-сервис общения.
- обычно поверх **HTTP/2**
- описание контрактов в `.proto`
- данные сериализуются **protobuf** (бинарно)

Почему `curl` не всегда достаточно:
- gRPC не выглядит как “простой HTTP/1.1 текст”
- но `curl` всё равно полезен для диагностики **до уровня приложения**: DNS, TCP, TLS, иногда HTTP/2 handshakes.

Практический вывод:
- множество проблем gRPC начинаются как обычные проблемы: DNS/TCP/TLS/HTTP2.

(Инструменты вроде `grpcurl` добавляют позже, когда базовые слои уже понятны.)

## 8. TLS/HTTPS

### Теория

**TLS** даёт:
- шифрование (конфиденциальность)
- целостность данных
- аутентификацию сервера сертификатом (обычно)

**Сертификат и доверие**
- сертификат сервера подписан CA
- проверяется цепочка: leaf → intermediate → root

**SAN и имя хоста**
- сертификат должен быть “действителен для этого имени” (SAN)

**SNI**
- когда на одном IP много сайтов, клиент сообщает нужное имя в handshake (SNI), иначе можно получить “не тот сертификат”.

Типовые ошибки:
- `certificate signed by unknown authority` — клиент не доверяет CA
- `certificate is not valid for requested host` — имя не совпало (SAN)

## 9. Firewall

### Теория

**Firewall / ACL / Security Group / Network Policy** — правила “кто кому может ходить”.
- бывают на хосте, в облаке, в k8s, в сервис-меше

Симптоматика:
- **refused**: порт закрыт / процесс не слушает / активный отказ (RST)
- **timeout**: пакеты “где-то” дропают / нет маршрута / firewall молча режет

Тебе не нужно сразу в iptables/nftables. Важно уметь:
- распознать симптом
- локализовать слой

## 10. NAT (почему “IP не тот”)

### Теория

**NAT** меняет адреса/порты на границе сети.
- поэтому “внутренний” IP может не совпадать с тем, что видит внешний мир

Почему это часто всплывает в Docker/K8s:
- внутри контейнера/пода один IP
- наружу выходит другой IP (хост/нода/балансер)

Если видишь `172.x.x.x` или `10.x.x.x` — это часто просто внутренняя сеть.

## 11. Инструменты

- `ip` — интерфейсы и маршруты
  - `ip a` (адреса)
  - `ip r` (маршруты)

- `ss` — сокеты и TCP состояния
  - `ss -tulpen` (слушающие)
  - `ss -tanp` (активные соединения)

- `curl` — HTTP/HTTPS диагностика
  - заголовки, редиректы, тайминги, детали соединения

- `dig` — DNS диагностика
  - имя → IP, TTL, какой резолвер отвечает

- `openssl` — TLS диагностика
  - сертификаты, цепочка, SNI, handshake

## 12. Диагностическая “лестница”

Когда “сервис недоступен”, иди так:

1) **DNS**: имя резолвится?
```bash
dig service.name
```

2) **TCP**: порт доступен? есть соединение?
- пробуй `curl -v http://host:port`
- смотри `ss -tanp` (есть ли SYN-SENT/ESTABLISHED)

3) **HTTP**: что возвращает сервер?
- коды/редиректы/заголовки
```bash
curl -v http://host:port
curl -I http://host:port
```

4) **TLS** (если https): сертификат/имя/SNI
```bash
openssl s_client -connect host:443 -servername host -showcerts
curl -v https://host
```

5) **Маршруты/NAT**: куда вообще идут пакеты?
```bash
ip r
ip a
```

6) Дальше — firewall/policies (следующий слой), но уже с пониманием, **где именно ломается**.
